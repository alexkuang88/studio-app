# 测试样例 / Test Scenario

## 测试目标

验证系统的核心逻辑：员工分段打单记录的金额自动计算、设备状态同步、工资计算、防呆机制。

## 测试数据

### 1. 创建订单

| 字段 | 值 |
|------|------|
| 订单号 | P001 |
| 订单来源 | 抖音客户 / Douyin |
| 下单时间 | 2026-05-29 07:00 |
| 要求完成时间 | 2026-05-30 07:00 |
| 目标金额 | 5000 万 |

### 2. 创建设备

| 设备编号 | 设备名称 | 初始状态 |
|----------|----------|----------|
| M001 | 1号机 | available |
| M002 | 2号机 | available |

### 3. 创建员工

| 员工编号 | 姓名 | 状态 |
|----------|------|------|
| N001 | 1号员工 | official |
| N002 | 2号员工 | official |
| N003 | 3号员工 | official |
| N004 | 4号员工 | official |

## 测试步骤

### Step 1: N001 开始打单（开始余额 0）

```
→ 操作: /entry/start-session
→ 选择订单: P001
→ 选择设备: M001
→ 选择打手: N001
→ 上号时间: 2026-05-29 07:00
→ 初始余额: 0
→ 点击"开始记录"

预期结果:
✅ work_session 创建成功，status = running
✅ 订单 P001 状态变为 in_progress
✅ 设备 M001 状态变为 in_use
```

### Step 2: N001 结束 → N002 接班

```
→ 操作: /entry/handover
→ 选择进行中设备: M001 — N001 — P001
→ 系统自动显示: 当前打手 N001、开始时间、开始余额 0
→ 填写结束时间: 2026-05-29 18:00
→ 填写结束余额: 1300
→ 系统自动计算:
   本次成绩 = 1300 - 0 = 1300 万
   工作时长 = 11 小时
   效率 = 1300 / 11 = 118.18 万/h
   
→ 选择接班打手: N002
→ 选择接班设备: M001（原设备，默认）
→ 系统自动设定:
   接班开始余额 = 1300
   接班开始时间 = 2026-05-29 18:00
→ 点击确认交接

预期结果:
✅ N001 的 work_session 状态变为 completed, result_amount = 1300
✅ 订单 P001 completed_amount = 1300
✅ N002 的 work_session 创建成功, status = running, start_amount = 1300
✅ 设备 M001 保持 in_use
✅ 交班余额自动传递，无断档
```

### Step 3: N002 结束 → N003 接班

```
→ 结束余额: 2700
→ 结束时间: 2026-05-30 06:00
  
自动计算:
   本次成绩 = 2700 - 1300 = 1400 万
   工作时长 = 12 小时（跨天）
   效率 = 1400 / 12 = 116.67 万/h

→ 接班: N003
→ 开始余额自动 = 2700
→ 开始时间自动 = 2026-05-30 06:00

预期结果:
✅ N002 成绩 = 1400 万
✅ 订单 P001 completed_amount = 1300 + 1400 = 2700
✅ N003 接班 start_amount = 2700（自动延续）
✅ 跨天时间计算正确（18:00 → 次日 06:00 = 12 小时）
```

### Step 4: N003 结束 → N004 接班

```
→ 结束余额: 3900
→ 本次成绩 = 3900 - 2700 = 1200 万

→ 接班: N004
→ 开始余额自动 = 3900
```

### Step 5: N004 结束（订单完成）

```
→ 结束余额: 5000
→ 结束时间: 2026-05-30 07:00
→ 本次成绩 = 5000 - 3900 = 1100 万

订单自动汇总:
  completed_amount = 1300 + 1400 + 1200 + 1100 = 5000

预期结果:
✅ 订单 completed_amount = 5000
✅ 目标达成: 5000 >= 5000 → 状态变为 ready_to_complete
✅ 系统提示: "已达到目标金额，可以完成订单"
```

### Step 6: 完成订单

```
→ 操作: /entry/complete-order
→ 选择订单: P001
→ 系统显示: 目标5000万、已完成5000万、剩余0万
→ 显示: "已达到目标金额，可以完成订单"
→ 点击"完成订单"

由于实际完成时间(2026-05-30 07:00) = 要求完成时间(2026-05-30 07:00)
预期结果:
✅ 按时完成 / Terminé à temps
✅ 设备 M001 自动变为空闲
✅ 订单状态 = completed
```

### Step 7: 验证员工月成绩

```
选择月份: 2026-05

预期结果:

| 1    | N002 | 1400万 | 12h    | 9800 Ar           |
| 2    | N001 | 1300万 | 11h    | 9100 Ar           |
| 3    | N003 | 1200万 | 10h    | 8400 Ar           |
| 4    | N004 | 1100万 | —      | 7700 Ar           |
```

### Step 8: 验证防呆机制

| 测试项 | 预期结果 |
|--------|----------|
| 同一订单同时两条 running | ❌ 拒绝 |
| 同一设备同时两条 running | ❌ 拒绝 |
| 同一打手同时两条 running | ❌ 拒绝 |
| end_amount < start_amount | ❌ 拒绝 |
| end_time <= start_time | ❌ 拒绝 |
| 维修中设备被选择 | ❌ 拒绝 |
| 已完成订单继续录入 | ❌ 拒绝 |
| 工资锁定后 Operator 修改 | ❌ 拒绝 |
| 作废不填原因 | ❌ 拒绝 |
| 接班余额自动传递 | ✅ 自动等于上一位结束余额 |

### Step 9: 验证设备状态同步

| 时间点 | M001 状态 | 说明 |
|--------|-----------|------|
| N001 开始打单后 | in_use | 被 N001 占用 |
| N001 结束, N002 接班 | in_use | 设备不释放，继续使用 |
| N004 结束后 | in_use | 仍被 N004 占用 |
| 完成订单 P001 后 | available | 自动释放 |
| M001 使用中时尝试给其他订单/打手 | ❌ | 拒绝重复占用 |

## 工资计算验证公式

```
工资 = 总成绩 / 100 × 工资单价

默认单价: 700 Ar / 100万

N001: 1300 / 100 × 700 = 9100 Ar  ✅
N002: 1400 / 100 × 700 = 9800 Ar  ✅
N003: 1200 / 100 × 700 = 8400 Ar  ✅
N004: 1100 / 100 × 700 = 7700 Ar  ✅
```
